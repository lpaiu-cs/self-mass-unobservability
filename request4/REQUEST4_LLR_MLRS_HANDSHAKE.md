# Request 4: MLRS Handshake Audit

## Scope

This stage is a **short, strict hand-off experiment** on top of the public
`MLRS Lunar Code` bundle.

It is not a weak-field fit. It asks a narrower question:

- can the public MLRS lunar sample workflow be made to replay on this macOS
  workspace without turning into a second bespoke estimator project,
- and if so, where is the nearest gravity-model insertion point for a later
  `delta_SEP` remapping.

## Why This Exists

After the APOLLO-only surrogate branch hit its stop rule, Request 4 needed a
canonical hand-off path rather than more surrogate growth.

The prior pivot scout already established that:

- CRD format inputs are publicly documented and partially public,
- Earthdata gating blocks live CDDIS fetches but not the public ILRS support
  tarballs,
- and `MLRS Lunar Code` is the closest public LLR-oriented stack in reach.

This audit is the next bounded step after that scout.

## Implementation

The audit summary is generated by
[request4_llr_mlrs_handshake_audit.py](request4_llr_mlrs_handshake_audit.py#L1).

The underlying lab copy lives in
`data/request4_llr/mlrs_handshake_lab/` and was used only as a bounded
modernization sandbox.

The hand-off experiment did four concrete things:

1. rebuilt the sample-critical binaries for local arm64 execution:
   - `llr_npt_crd`
   - `Poisson_crd`
   - `lun_auto_crd`
   - `lun_sdqi_crd`
   - `frd_strip`
2. linked `llr_npt_crd` against a local `PLEPH` library extracted from the
   public JPL `testeph1.f`,
3. added the public DE421 Linux binary ephemeris
   `lnxp1900p2053.421` as `data/pred/jpleph.421` and exposed it through the
   top-level `JPLEPH` symlink expected by the JPL reader,
4. silenced the informational `PLEPH` banner in
   [pleph_lib.f](../data/request4_llr/mlrs_handshake_lab/src/llr_npt/pleph_lib.f#L713)
   so MLRS `.errorc` files keep their original "failure-channel" semantics.

## Current Result

The MLRS sample replay now works end-to-end in the lab copy.

For the two bundled public lunar sample cases:

- both `.frd` outputs match the reference files **exactly**,
- both `.npt` outputs differ from reference in exactly **one line** each,
- and those `npt` drifts are small final-line value shifts rather than gross
  replay failure.

In other words, the MLRS path is no longer blocked by:

- Linux-only distributed binaries,
- missing `JPLEPH`,
- or immediate toolchain incompatibility.

It has crossed the sample-reproduction gate.

## Pipeline Interpretation

The public batch script
[ldb_crd](../data/request4_llr/mlrs_handshake_lab/bin/ldb_crd#L115)
already exposes a clean three-stage structure:

1. `llr_npt_crd -r`:
   prediction and residual recalculation from `.frcal` to `.frrec`
2. `Poisson_crd`, `lun_auto_crd`, `lun_sdqi_crd`:
   filtering and cleanup
3. `llr_npt_crd -n`:
   normal-point construction from `.frfil3` to `.frfin/.npt`

The nearest gravity-model bridge sits in
[jjreadnp.f](../data/request4_llr/mlrs_handshake_lab/src/llr_npt/jjreadnp.f#L5),
which explicitly states that it links the normal-point/prediction programs to
the JPL `PLEPH` reader and evaluates the model states through those calls.

So, within the current public MLRS stack, the natural EFT hook is the
`recalc/prediction` layer, not the downstream filtering layer.

## Decision

This audit changes the Request 4 pivot in one important way:

- `MLRS` should no longer be treated as a speculative legacy code path.
- It is now a **live hand-off candidate** because the public sample workflow
  replays on this machine with near-reference outputs.

That does **not** mean Request 4 is solved.

It still means:

- no weak-field posterior exists yet,
- no nuisance-estimation stack has been validated here,
- and this MLRS branch should remain bounded rather than growing into another
  bespoke estimator project.

The correct reading is narrower:

- the APOLLO-only bespoke branch failed its stop rule,
- the MLRS handshake passed its sample-reproduction gate,
- so the remaining Request 4 question is now whether this public MLRS path can
  be used as the bounded estimator hand-off layer for a later EFT remapping.

That follow-on question is now addressed one step further in
`REQUEST4_LLR_MLRS_INTERFACE_PROBE.md` and
`REQUEST4_LLR_MLRS_RECALC_SEAM_PROBE.md`: the recalc layer exposes a bounded
local correction seam, and that seam behaves linearly under small synthetic
probes, but the project still needs to decide whether the real `delta_SEP`
physics fits through the deeper prediction/state seam without expanding into a
broader legacy-code restoration project.
